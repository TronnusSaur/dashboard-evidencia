Prompt para otra instancia de ChatGPT
Necesito implementar una optimización incremental en este repositorio para evitar que el sistema vuelva a escanear masivamente Google Drive y Google Sheets cada vez que se actualiza la auditoría.

Contexto actual del proyecto
El sistema actual usa un script batch ubicado en scripts/auditData.js. Ahí se configuran las etapas E1 y E2, cada una con una carpeta maestra de Google Drive.

El script autentica contra Google usando googleapis y actualmente usa scopes de solo lectura para Drive y Sheets. El proyecto ya tiene la dependencia googleapis en package.json.

La función obtenerPaginado hace llamadas drive.files.list paginadas para listar archivos/carpetas en Drive.

La función auditarFotos recibe IDs de carpetas de folios y por cada carpeta hace otra llamada a drive.files.list para listar fotos y detectar si existen imágenes con patrones _inicial, _caja y _terminado.

El flujo principal por etapa hace tres cosas pesadas:

Mapea carpetas de contrato y folio desde Drive.

Lee registros desde Google Sheets con sheets.spreadsheets.values.get.

Audita cada folio contra Drive y usa un cache limitado que solo evita repetir folios ya marcados como OK.

Después de procesar todos los datos, el script borra los JSON públicos anteriores y regenera los archivos de salida en public/contratos, además de regenerar src/dataMock.js.

El frontend no llama directamente a Google Drive ni a Google Sheets. Consume datos estáticos desde src/dataMock.js y archivos JSON bajo /contratos/.... Para cargar el detalle, el dashboard hace fetch a rutas como /contratos/${stage}_${empresa}_${contrato}.json.

src/dataMock.js es un archivo autogenerado con ERROR_TYPES, GLOBAL_TOTALS, RESUMEN_DATA y FILTERS_MAP.

Objetivo de la optimización
Convertir el sistema de:

escaneo completo de Drive + lectura completa de Sheets + auditoría de miles de folios + regeneración total
a:

evento de cambio + lectura incremental del cambio + recálculo solo del folio/contrato afectado + actualización parcial de JSON/resúmenes
La meta realista no es “una llamada API total en todos los escenarios”, sino reducir miles de llamadas a:

1 webhook entrante
+ 1 llamada a changes.list
+ pocas llamadas adicionales solo si faltan metadatos o si el índice local no conoce el archivo/carpeta
La documentación oficial de Google Drive indica que las notificaciones push sirven para reducir costos de polling y cómputo, pero también aclara que las notificaciones no contienen el detalle del cambio; solo avisan que hay cambios disponibles, por lo que se debe consultar el feed de cambios con changes.list.
Fuentes oficiales:

https://developers.google.com/workspace/drive/api/guides/push

https://developers.google.com/workspace/drive/api/guides/manage-changes

5. Implementar procesamiento incremental de Drive
Crear:

scripts/incremental/changeProcessor.js
Debe exponer una función principal:

export async function processDriveChanges({ drive, sheets, stateStore }) {
  // 1. Leer último pageToken guardado.
  // 2. Llamar drive.changes.list.
  // 3. Procesar páginas hasta newStartPageToken.
  // 4. Deducir folios afectados.
  // 5. Recalcular solo esos folios.
  // 6. Actualizar índices, agregados y JSON afectados.

Lógica esperada
Por cada cambio de Drive:

Tomar fileId.

Buscarlo en drive_items.

Si existe:

obtener stage, empresa, contrato, folio;

marcar ese folio como afectado.

Si no existe:

consultar drive.files.get con campos mínimos:

id,name,mimeType,parents,trashed,modifiedTime
intentar resolver el folio subiendo por parents usando el índice.

Si es un archivo de foto:

detectar categoría por nombre (_inicial, _caja, _terminado).

Si es carpeta de folio:

actualizar folderId -> folio.

Recalcular solo los folios afectados.

7. Actualizar agregados de forma diferencial
Hoy el sistema recalcula globalTotals, errorTypesSet, filtersMap y resumenData desde todos los datos.

En incremental debe hacerse así:

estado anterior del folio -> restar 1 del contador correspondiente
estado nuevo del folio -> sumar 1 al contador correspondiente
Ejemplo
Si un folio pasa de:

FALTA: CAJA
a:

OK
Actualizar:

GLOBAL_TOTALS["FALTA: CAJA"] -= 1
GLOBAL_TOTALS["OK"] += 1

RESUMEN_DATA[empresa, contrato, stage]["FALTA: CAJA"] -= 1
RESUMEN_DATA[empresa, contrato, stage]["OK"] += 1
Además:

si el folio ahora es OK, removerlo del JSON público del contrato;

si el folio ahora no es OK, insertarlo o actualizarlo en el JSON público del contrato.

Resumen ejecutivo
La implementación viable es:

No tocar el dashboard de forma profunda, porque ya consume archivos estáticos.

Convertir el script batch en módulos reutilizables, porque hoy todo el flujo pesado vive concentrado en scripts/auditData.js.

Crear un índice persistente de Drive/Sheets para no volver a descubrir 50k+ folios en cada corrida.

Usar Drive changes.watch + changes.list para recibir eventos y consultar deltas.

Actualizar solo el folio/contrato afectado y ajustar agregados por diferencia.

Mantener una reconciliación completa periódica como respaldo.
